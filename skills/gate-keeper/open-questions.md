# Open Questions

當需求仍有關鍵缺口時，使用以下格式：

```markdown
## Open Questions

- OQ-01：[問題描述] → 待確認對象：[PM / SA / 後端 / 使用者]
- OQ-02：[問題描述] → 待確認對象：[PM / SA / 後端 / 使用者]
```

## 分級原則

- 可於 Step 1 一併整理，不阻擋 spec 撰寫
- 若問題會影響商業規則、API contract、資料流或測試預期，則不得進入 `/plan`
- 不影響核心邏輯者，可標記待確認並先往下進行

## 禁止事項

- 不得自行補齊未確認的商業規則
- 不得用模糊句子取代具體問題
- 不得將可阻擋開發的缺口只寫成備註而不進入 OQ
