# 規格文件撰寫規範

| 檔案 | 用途 |
|------|------|
| [ears-syntax.md](ears-syntax.md) | EARS 語法詳細說明與範例 |
| [templates.md](templates.md) | 各頁面類型規格範本（查詢頁/明細頁/對話框） |

## 適用情境

- 為新頁面/功能建立規格文件
- 更新既有規格（需求變更、測試結果記錄）
- 撰寫驗收條件（快樂路徑 + 錯誤路徑）
- spec-writer agent 撰寫 spec 時參考

## 核心規則

1. **MVP 優先**：只描述最小可交付範圍，不預設未確認功能
2. **可測試性**：每個規格必須包含 Given-When-Then 驗收條件
3. **AC 編號必填**：每條驗收條件必須使用 `AC-XX` 編號，方便後續測試追蹤
4. **EARS 語法**：行為描述使用 EARS 格式，確保無歧義且可驗證
4. **正體中文**：所有規格文件一律使用正體中文，禁止簡體
5. **機敏資訊防護**：禁止包含真實 IP、憑證、個資
6. **不重複定義**：已在基底類別實作的行為（分頁、catchError、Session 暫存）不重複寫入 spec
7. **手動測試步驟**：每份 spec 必須包含此章節，不可省略
8. **商業規則先寫**：會影響驗收、API contract 或測試預期的規則，必須先寫在 `商業規則` 章節
9. **Step 2 可規劃**：spec 內容需足以支撐後續 plan，至少要能映射測試層級、資料流與 E2E 決策

## 規格結構（必要章節）

1. **功能名稱** — 一句話描述
2. **功能描述 / 使用者故事** — 商業背景、使用情境與價值
3. **商業規則** — 明確條件、限制、計算邏輯、例外
4. **畫面規格** — 路由、區塊描述、欄位清單、表格欄位
5. **畫面行為規格（EARS 語法）** — 查詢區、結果區、勾選/匯出
6. **API 規格** — 端點、Request/Response 介面、錯誤情境
7. **驗收條件** — `AC-XX` + Given-When-Then（含快樂路徑、邊界、空值、異常與 Loading/Error 狀態）
8. **Step 2 計畫輸入** — 測試層級規劃、資料流、E2E 決策
9. **手動測試步驟** — 場景、前置條件、操作步驟、預期結果
10. **技術實作備註**（可選） — 基底類別、覆寫方法、特殊邏輯

## 驗收條件格式

```markdown
- [ ] AC-01
	Given [前置條件]
	When  [使用者操作]
	Then  [預期結果，須具體可量化]

- [ ] AC-02
	Given [邊界 / 錯誤 / 空值前置條件]
	When  [觸發條件]
	Then  [預期錯誤處理、保留狀態或提示內容]
```

## 禁止事項

- 不在規格中定義尚未確認的功能
- 不預設「未來可能需要」的擴充欄位
- 不重複定義已在基底類別中實作的行為
- 不使用模糊的驗收條件（如「系統應正常運作」）
- 不省略 `商業規則`、`AC-XX` 或 Step 2 所需的測試 / 資料流 / E2E 判斷
