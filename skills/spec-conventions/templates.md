# 頁面類型規格範本

## 通用規格骨架

每份 spec 至少應包含以下章節：

```markdown
## 功能：[功能名稱]

### 功能描述
（說明功能目的、商業背景、使用者情境）

### 商業規則
- 規則 1
- 規則 2

### 畫面 / 模組規格
（路由、區塊、欄位、表格、按鈕）

### 畫面行為規格（EARS）
- When ... , the system shall ...
- If ... , then the system shall ...

### API 規格
- 查詢 API：
- 異動 API：
- 錯誤碼 / 錯誤處理：

### Acceptance Criteria

- [ ] AC-01
  Given [前置條件]
  When  [觸發動作]
  Then  [預期結果]

- [ ] AC-02
  Given [邊界 / 空值 / 異常條件]
  When  [觸發動作]
  Then  [錯誤處理 / 提示 / 狀態保留]

### Step 2 計畫輸入

#### AC 對應測試層級

| AC | 測試層級 | 原因 |
|---|---|---|
| AC-01 | Service Unit / Component / E2E | [說明] |

#### 資料流

- API DTO -> Domain Model -> Form Model -> Request DTO
- 後端錯誤碼對應處理

#### E2E 決策

- [ ] 需要 Playwright E2E
- [ ] 可略過 E2E，理由：

### 非功能需求
- 效能：
- 資安：
- 稽核：

### 手動測試步驟

| # | 測試場景 | 驗證重點 |
|---|---------|---------|

### 測試紀錄

| 日期 | 測試類型 | 通過率 | 結論 | 備註 |
|------|---------|--------|------|------|

### 變更歷程

| 日期 | 版本 | 變更內容 | 變更者 |
|------|------|---------|--------|
```

## 查詢頁面附加規格

查詢頁面除基本結構外，額外須定義：

- **基底類別選擇**：專案的查詢基底類別（若有提供含勾選功能的變體，需一併說明）
- **分頁行為**：每頁筆數預設值
- **勾選功能**（若適用）：
  - 唯一識別欄位（用於 `getItemId()` 或同等方法）
  - 是否需條件式勾選（覆寫可選過濾方法）
- **Excel 匯出**（若適用）：匯出表頭、資料格式化
- **查詢條件保留**：是否設定查詢條件暫存機制（如 Session/URL 暫存）
- **基底類別對應**：
  - Request 參數轉換邏輯
  - 呼叫的查詢 API
  - 需覆寫的可選方法

### 查詢頁面手動測試步驟（必須涵蓋）

| # | 測試場景 | 驗證重點 |
|---|---------|---------|
| 1 | 查詢成功 | 填入有效條件 → 按查詢 → 表格顯示結果 |
| 2 | 查詢無結果 | 填入無匹配條件 → 按查詢 → 顯示無資料提示 |
| 3 | 必填未填 | 清空必填欄位 → 按查詢 → 顯示欄位錯誤提示 |
| 4 | 分頁切換 | 查詢多頁結果 → 切換至第 2 頁 → 結果更新 |
| 5 | 勾選跨頁（若適用） | 第 1 頁勾選 → 切換第 2 頁 → 回第 1 頁 → 勾選維持 |
| 6 | Excel 匯出（若適用） | 勾選項目 → 按匯出 → 下載檔案並驗證內容 |

### 查詢頁面 Step 2 計畫輸入補充

- 查詢 API DTO 與表格顯示 Domain Model 的映射
- 查詢條件 Form Model 與 Request DTO 的轉換
- 匯出 / 勾選 / 跨頁選取由哪一層驗證
- 若結果頁涉及關鍵旅程，需標示是否進入 E2E

## 明細/編輯頁面附加規格

- **返回行為**：返回路由、是否傳遞 `returnUrl`
- **離開提示**：是否實作離開確認介面、快照哪些資料
- **快照型別**：快照型別需明確定義，不可使用 `any`
- **唯讀模式**：進入條件、隱藏按鈕
- **提交後行為**：導航目標、快照同步、成功提示訊息
- **自訂錯誤處理**：是否需攔截業務錯誤並自訂提示

### 明細頁面手動測試步驟（必須涵蓋）

| # | 測試場景 | 驗證重點 |
|---|---------|---------|
| 1 | 載入資料 | 從查詢頁點擊 → 正確載入明細資料 |
| 2 | 提交成功 | 修改欄位 → 按提交 → 顯示成功提示 + 導航 |
| 3 | 離開未儲存 | 修改欄位 → 未提交即離開 → 確認對話框出現 |
| 4 | 唯讀模式 | 以唯讀參數進入 → 編輯按鈕隱藏、欄位不可編輯 |
| 5 | 返回查詢頁 | 按返回 → 回到查詢頁且查詢條件保留 |

### 明細/編輯頁面 Step 2 計畫輸入補充

- 載入資料與提交資料的 DTO / Domain / Form 轉換
- 業務錯誤碼如何映射到欄位錯誤或頁面提示
- 哪些 AC 由 Component Test 驗證，哪些需進 E2E

## 對話框元件附加規格

- **開啟方式**：觸發來源、Modal 設定（size、backdrop 等）
- **查詢功能**：查詢 API、錯誤處理（catchError 清空結果）
- **回傳資料**：回傳事件的回傳型別
- **關閉行為**：關閉時機、是否清空狀態

### 對話框手動測試步驟（必須涵蓋）

| # | 測試場景 | 驗證重點 |
|---|---------|---------|
| 1 | 開啟對話框 | 點擊觸發按鈕 → 對話框正確開啟 |
| 2 | 查詢並選取 | 輸入條件查詢 → 選取結果 → 回傳值正確帶入父元件 |
| 3 | 查詢無結果 | 輸入無匹配條件 → 顯示無資料提示 |
| 4 | 關閉對話框 | 按關閉/取消 → 對話框關閉、父元件無異動 |

### 對話框元件 Step 2 計畫輸入補充

- 查詢結果與回傳資料型別如何映射
- 關閉、取消、確認分別由哪個測試層級驗證
- 若對話框只影響單頁互動，通常可不新增 E2E，但需在 spec 記錄理由

## 測試紀錄格式

spec 文件底部需記錄測試結果：

```markdown
## 測試紀錄

| 日期 | 測試類型 | 通過率 | 結論 | 備註 |
|------|---------|--------|------|------|
| {date} | Playwright E2E | X% | 通過/不通過 | {notes} |
```

## 變更歷程格式

```markdown
## 變更歷程

| 日期 | 版本 | 變更內容 | 變更者 |
|------|------|---------|--------|
| {date} | 1.0.0 | 初版建立 | spec-writer |
```
