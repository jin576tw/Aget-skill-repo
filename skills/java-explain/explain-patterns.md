# Java 解釋模板與範例

## 零、學習者程度設定

**目標對象**：完全沒有 Java 背景、正在學習 adp-policy 程式碼的保險業務人員。

**解說原則**：
- 禁止直接使用 Java 術語，**每個術語必須附白話括弧**，例如：「Annotation（標記）」
- 比喻優先：辦公室、信封、翻譯員、收發室等保險業熟悉的場景
- 粒度：以「段落」為單位（每個 method 一段），不逐行翻譯語法
- 長度控制：單一方法說明不超過 5 句；整個 class 解說不超過 400 字（複雜類別除外）
- 每次解說結尾**必須**附上「理解確認題」（見第六節）

## 一、類別角色判斷表

| 類別名稱特徵 | 角色 | 白話說明 |
|---|---|---|
| 結尾是 `Controller` | 收發室 | 負責接收前端的請求、送出回應，自己不處理業務 |
| 結尾是 `Service` | 業務員 | 真正處理業務邏輯的地方（計算、判斷、組合資料）|
| 結尾是 `Repository` | 資料庫窗口 | 負責從資料庫撈資料或存資料，不做任何業務判斷 |
| 結尾是 `Entity` | 資料庫表的鏡像 | 對應資料庫的一張表，每個欄位就是表的一個欄位 |
| 結尾是 `Dto` / `Vo` / `Cmd` | 資料信封 | 在不同地方傳遞資料用的容器，像信封一樣裝著內容 |
| 結尾是 `Mapper` | 翻譯員 | 把 Entity 轉成 Dto，或反過來，不做計算 |
| 結尾是 `Config` | 環境設定檔 | 告訴 Spring 要怎麼啟動、連線哪個資料庫等 |
| 結尾是 `Api` | 菜單 / 合約 | 規定這個收發室要提供哪些服務，具體怎麼做由 Controller 決定 |

## 二、常見 Java 語法白話對照表

| Java 語法 | 白話意思 |
|---|---|
| `@RestController` | 標記這個類別是「收發室」|
| `@GetMapping("/path")` | 當前端用 GET 訪問 /path 這個網址時，呼叫這個方法 |
| `@PostMapping("/path")` | 當前端用 POST 送資料到 /path 時，呼叫這個方法 |
| `@Autowired` / 建構子注入 | 跟 Spring 說「我需要這個工具，幫我準備好」|
| `@Service` | 標記這個類別是「業務員」|
| `@Repository` | 標記這個類別是「資料庫窗口」|
| `@Entity` | 標記這個類別對應一張資料庫表 |
| `@Transactional` | 這個操作要一起成功或一起失敗（例如：轉帳要存款和扣款同時完成）|
| `@Override` | 我正在履行 interface（菜單）上的承諾，這不是我自己發明的方法 |
| `@Slf4j` | 幫我產生 log 工具，可以記錄程式執行時的訊息 |
| `@RequiredArgsConstructor` | 幫我自動把 final 欄位的工具注入進來，不用手寫建構子 |
| `implements XxxApi` | 承諾「我會做 API 菜單上的所有功能」 |
| `Optional<T>` | 回傳的結果「可能有，也可能沒有」，避免程式崩潰 |
| `List<T>` | 一個裝了多筆同類型資料的清單 |
| `void` | 這個方法執行完不回傳任何東西 |
| `final`（在欄位） | 這個工具一旦拿到就不能換掉（由 Spring 注入） |
| `private` | 只有這個 class 自己能用，外面看不到 |
| `public` | 對外開放，外面可以呼叫 |
| `throws Exception` | 這個方法如果出問題，會拋出錯誤讓呼叫者處理 |

## 四、範例：ReceiptController 解說

```
## ReceiptController 解說

**角色**：收發室（負責接收前端請求、送出回應）

**一句話總結**：接收前端發來的「列印收據」、「查詢收據」等請求，轉交給 ReceiptService
處理，再把結果送回前端。

**對應業務文件**：
`adp-knowledge/收據作業/收據管理作業/收據列印管理/FLOWCHART.md`

---

### 各方法說明

**printReceipt()**
- 觸發時機：使用者在畫面上點「列印」按鈕
- 做了什麼：把前端送來的收據編號交給 ReceiptService，ReceiptService 去資料庫找資料後回傳
- 回傳什麼：收據資料（PDF 路徑或收據內容）

**queryReceipt()**
- 觸發時機：使用者輸入查詢條件後按「查詢」
- 做了什麼：把查詢條件（日期範圍、保單號碼等）傳給 Service，取回符合條件的收據清單
- 回傳什麼：收據清單

---

### 下一步建議

學完這個類別後，建議接著看：
- `receipt/service/ReceiptService.java` — 業務員，真正決定收據怎麼組出來的地方

---

### ✏️ 理解確認題（請回答後我會批改）

Q1. ReceiptController 這個 class 的角色是什麼？它自己會去資料庫撈資料嗎？

Q2. `printReceipt()` 前面如果有 `@Override`，代表什麼意思？

Q3. 如果今天要改「列印收據的格式」，你覺得應該改 Controller 還是 Service？為什麼？
```

## 五、學習路線建議

### 適合初學者的學習順序（以「核保審核」為例）

```
1. FLOWCHART.md（業務流程圖）
   ↓ 了解流程後
2. XxxController（收發室）→ 看有哪些入口
   ↓
3. XxxService（業務員）→ 看它怎麼實現業務
   ↓
4. XxxRepository（資料庫窗口）→ 看它怎麼查資料
   ↓
5. XxxEntity（資料表）→ 看資料長什麼樣
```

### 常見 Java 概念學習對應

| 想理解的概念 | 在 adp-policy 看哪裡 |
|---|---|
| 什麼是 Spring Boot | `common/config/` 下的 Config 類別 |
| 什麼是依賴注入 | 任何 Service 的建構子（`@RequiredArgsConstructor`）|
| 什麼是 JPA / 資料庫操作 | `*Repository.java` 類別 |
| 什麼是 REST API | `*Controller.java` 的 `@GetMapping`、`@PostMapping` |
| 什麼是例外處理 | `common/exception/` 或 Service 中的 try-catch |
| 什麼是 DTO 模式 | `*Cmd.java`、`*Vo.java`、`*Dto.java` |
| 什麼是 interface / implements | 任何 `implements XxxApi` 的 Controller |
| 什麼是 Drools 規則引擎 | `resources/rules/gi/nb/insured/nb-insured.drl` |
| 什麼是串流操作 stream() | Controller 或 Service 中的 `.stream().filter().map()` |

## 六、理解確認題格式規範

每個 session 解說完畢後**必須出至少 10 題**，分批或一次出完皆可，但一個 session 總題數不得少於 10 題。題型如下，難度對應學習者的「零 Java 背景」程度：

### 題型分配（10 題範例）

| 題號 | 難度 | 題型 | 範例 |
|---|---|---|---|
| Q1–Q3 | 基礎（記憶） | 角色/定義確認 | 「XXX 的角色是什麼？」「`final` 代表什麼？」 |
| Q4–Q6 | 應用（理解） | 場景判斷 | 「如果要改 XXX，你應該看哪個 class？」 |
| Q7–Q8 | 進階（分析） | 比較/選擇 | 「DTO 和 Cmd 有什麼不同？什麼時候用哪個？」 |
| Q9–Q10 | 延伸（思考） | 開放推論 | 「為什麼 Service 不能呼叫 Controller？」「如果少了這一行會怎樣？」 |

### 批改規則（Claude 看到使用者回答後執行）

**嚴格模式：答案必須包含關鍵詞才算正確；模糊或不完整的回答不放行。**

#### 判斷標準

| 等級 | 條件 | 行動 |
|---|---|---|
| ✅ 完全正確 | 包含關鍵概念 + 說得清楚 | 給正面確認 + 一句補充深化知識 |
| 🔶 部分正確 | 方向對但缺關鍵詞，或說法太模糊 | 指出哪裡對、哪裡缺，給**線索**，要求重新回答 |
| ❌ 錯誤 | 概念錯誤或答非所問 | 給**線索**引導方向，要求重新回答（不直接給答案） |

#### 嚴格批改細則

- **「模糊說法」不算正確**：例如「做事情的地方」不如「業務邏輯的地方（計算、判斷、組合資料）」
- **缺少關鍵詞不放行**：每題出題時內心預設 1-2 個關鍵詞，沒說到就要補答
- **不接受「大概對」**：若使用者說對了結果但沒說出原因，要追問「為什麼」
- **補答次數不限**：使用者回答正確前，持續要求補答，不能因為「差不多」就放過

#### 答錯時的線索格式（重要）

答錯或部分正確時，**不直接給答案**，改用以下格式給線索：

```
🔍 線索：
  - 去看 [具體檔案路徑 + 行號]，找 [具體關鍵字或方法名]
  - 或：回想「[本次解說的某個比喻]」，XXX 的角色是什麼？
  - 或：想想看，如果 XXX 不做這件事，會發生什麼？
```

線索原則：
- 指向**可以找到答案的地方**（檔案、行號、比喻），不直接說出答案
- 最多給 2 條線索，讓使用者有空間自己推論
- 用問句引導而非陳述句（「你覺得…？」比「答案是…」更好）

#### 批改完後的動作

- 所有題都答對 → 問「要繼續看下一個 class，還是有哪個概念還不確定？」
- 有題目待補答 → 只列出「哪幾題需要補答」+ 給對應線索，等待回答，不繼續下一個主題
