# /start-goal — Goal 預檢 + 條件產生器

讀取 `$ARGUMENTS` 作為使用者的 goal 意圖描述，執行 5 項預檢，輸出修正後的完整 `/goal` 條件。

## 背景知識

**語法**：`/goal <condition>`

**evaluator 機制**：
- 每輪結束由小型快速模型（預設 Haiku）讀取 Claude 的對話輸出，判斷條件是否達成
- evaluator **不執行指令、不讀檔案**，只能根據 Claude 已在對話裡呈現的內容判斷
- 條件必須能被「輸出文字」所驗證，不能只靠 Claude 口頭說「完成了」

**條件寫作框架**：
```
[工具/agent] 對 [範圍] 做 [動作]，最後一次輸出顯示 [具體結果]，且 [限制條件]，或 N 輪後停止
```

**限制**：
- 最低版本需求：Claude Code v2.1.139+
- `disableAllHooks` 或 `allowManagedHooksOnly` 未被關閉（evaluator 是 hooks 系統的一部分）

## 執行步驟

1. 分析 `$ARGUMENTS` 的內容
2. 對照 5 項檢查清單逐一評分
3. 輸出評分結果表格
4. 自動產出修正後的完整 `/goal` 條件供複製

## 檢查清單（5 項）

| # | 項目 | 判斷標準 |
|---|------|---------|
| 1 | **工具/agent** | `❌` 未指定工具（evaluator 無依據）；`✅` 有明確工具或 agent（如 `test-writer`、`/todo`） |
| 2 | **任務範圍** | `❌` 完全模糊；`⚠️` 範圍不明確；`✅` 有具體動作與範圍（如「針對 AC-01~05」） |
| 3 | **具體結果** | `❌` 沒有可驗證輸出；`⚠️` 描述模糊；`✅` 具體可核對文字（如「輸出顯示『未完成項目：0』」） |
| 4 | **限制條款** | `⚠️` 未說明（多輪推進後範圍可能跑掉）；`✅` 明確限制（如「不修改 spec 檔案」） |
| 5 | **停損子句** | `❌` 完全沒有（條件若不可能成立則無限燒 token）；`✅` 有明確輪數上限（如「或 20 輪後停止」） |

## 輸出格式

輸出以下結構：

```
## /goal preflight 結果

| 項目 | 狀態 | 說明 |
|------|------|------|
| 工具/agent | ✅/⚠️/❌ | ... |
| 任務範圍 | ✅/⚠️/❌ | ... |
| 具體結果 | ✅/⚠️/❌ | ... |
| 限制條款 | ✅/⚠️ | ... |
| 停損子句 | ✅/❌ | ... |

**結論**：（全部通過 / 有 N 項需修正）

**修正後條件**（可直接複製）：
```
/goal ...完整條件，含工具、範圍、具體結果、限制、停損子句...
```
```

若有 ❌ 項目，在修正後條件中自動補齊缺少的欄位（推斷合理的 agent、補上停損子句等）。
若僅有 ⚠️，仍輸出條件但標注建議改善處。
