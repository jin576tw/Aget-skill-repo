# /print-sd

依固定風格產生一份**工程師可直接開發**的 SD 實作規格書（Markdown），並跑完評分回圈。

用法：`/print-sd <需求描述或需求文件路徑>`

## 用途

把「需求 → 可施工的規格書」這段流程標準化。適用於：新功能規格、既有頁面的功能增量（feature delta）、需與後端對規格的預先設計。

**不適用**：純 bug fix（用 `error-first-debug`）、程式碼分析（用 code-analysis 工具鏈）。

---

## 撰寫風格規範（8 條，spec-writer 必須遵守）

> ⚠️ **這 8 條會覆寫 `spec-writer` agent 內建的預設規範**。派工時必須把本節全文貼給 spec-writer，並明確聲明「以下規則覆寫你的預設章節結構」，否則 spec-writer 會照它自己的預設寫出 AC-XX 編號與 E2E 章節。

| # | 規範 | 具體要求 |
|---|------|---------|
| 1 | **表格優先** | 除說明性敘述外，**長內容與步驟一律用表格**。條列（`- ` / `1. `）僅允許用於 3 項以內的短清單或單一路徑引用。業務規則、元件介紹、欄位清單、API 端點、錯誤情境、技術備註，全部表格化 |
| 2 | **施工閘門表** | 文件開頭（功能描述之後）必備「## 施工閘門（開發前必讀）」表，欄位為：開發項目／規格成熟度／可否進入開發／阻斷原因·前置條件。依成熟度分流為「可開發 ✅／需先完成查證 ⚠️／暫緩開發 ⛔」，並在被標為暫緩的章節就地加上 ⛔ blockquote 標示 |
| 3 | **EARS 表格化** | 「畫面行為規格」用 EARS 語意但以**表格**呈現，欄位為：#／語法（When·While·If-then）／觸發條件／系統行為（the system shall）／補充說明 |
| 4 | **驗收與測試整併** | **不分開寫「驗收條件」與「手動測試步驟」**，整併為單一「## 驗收與測試」章節。先給「測試項目總覽」表（#／測試目標／對象／測試層級），再逐項給小表（前置條件／操作步驟／預期結果／驗證重點） |
| 5 | **不用 AC-XX 編號** | 測試項目用流水號 `1. 2. 3.` 加**具描述性的標題**（如「206 — 新增指定指數帳戶列」），避免讀者需先釐清 AC 編號代表什麼 |
| 6 | **不產生 E2E 章節** | 不寫「E2E 決策」「Step 2 計畫輸入」等章節。測試層級直接寫在測試總覽表的欄位裡（Component Test／手動／施工前查證） |
| 7 | **開放問題表** | 必備「## 開放問題與施工前查證事項」表，欄位為：#／待確認項目／目前狀態／需確認對象／未確認的風險。內文引用時一律寫「見『開放問題與施工前查證事項』第 N 項」，**確保該項次真的存在** |
| 8 | **變更歷程 + 行號核實** | 末尾必備「## 變更歷程」表，欄位為**日期／版本／變更內容／變更者**（4 欄）。⚠️ 此格式**覆寫 spec-writer 內建的 3 欄規則**（日期／需求單號／變更說明，其文件標示「不得增刪欄位」），派工時須明講。**規格中每一處檔案路徑、行號、方法名、欄位名都必須實際開檔核實**，不得憑記憶或推測填寫 |

### 額外硬性要求

| 項目 | 要求 |
|------|------|
| 語言 | 正體中文，禁止簡體 |
| 型別 | 禁止 `any`，所有型別明確定義 |
| 推測內容 | 凡非需求文件明文的設計，一律標示為「暫定／推測」並列入開放問題表 |
| 機敏資訊 | 真實 IP、憑證、個資一律遮蔽 |
| 既有能力 | 不重複定義基底類別已實作的行為，改以「不需自行實作」+ 出處說明帶過 |

## 執行流程（三段）

### 第 1 段：spec-writer 撰寫

| 步驟 | 動作 |
|------|------|
| 1.1 | 確認需求來源（需求文件／BSD／Jira／使用者口述）與目標 spec 路徑（慣例：`specs/feature/{代號}/spec.md`） |
| 1.2 | **先自行探勘原始碼**：用 Glob/Grep 定位目標元件、基底類別、Service、型別檔，記下真實行號 |
| 1.3 | 派工 `spec-writer`，prompt 須包含：① 需求摘要 ② **本 command「撰寫風格規範」8 條全文** ③ 明確聲明「這 8 條覆寫你的預設章節結構，不要產生 AC-XX 編號、不要 E2E 章節、不要把驗收條件與手動測試步驟分開，**變更歷程改用 4 欄格式（日期／版本／變更內容／變更者），覆寫你內建的 3 欄需求單號規則**」 ④ 步驟 1.2 探勘到的檔案路徑與行號 ⑤ 目標輸出路徑 |
| 1.4 | spec-writer 回傳信號後，主會話**自行完整讀過產出的 spec.md**，確認風格 8 條到位 |

### 第 2 段：spec-reviewer 評分回圈

| 步驟 | 動作 |
|------|------|
| 2.1 | 派工 `spec-reviewer`，prompt 須包含：① spec.md 路徑 ② 應核對的原始碼檔案清單 ③ **使用者豁免項聲明**（明寫「移除 E2E 章節／移除 AC-XX 編號／驗收條件與手動測試步驟整併」是使用者明確要求，不得以憲法範本為由扣分）④ **本 command「撰寫風格規範」8 條全文**，作為其 rubric「C. 風格合規（20 分）」的計分依據——不附全文，審查者會誤用 `spec-conventions` skill 的舊格式（AC 編號必填、獨立手動測試步驟章節）當基準，與豁免聲明互相矛盾 |
| 2.2 | 取得 `spec-review` 信號行與分數，**原文轉述浮上對話** |
| 2.3 | **< 80 分**：把「必須補齊清單」中屬於「撰寫缺陷」的項目交回 `spec-writer` 改寫，回到 2.1 重審。**回 spec-writer 時必須重新附上 8 條風格規範全文 + 覆寫聲明，不可只傳缺陷清單**（subagent 每次都是全新 context，不附全文就會退回預設格式）。**最多 3 輪**，仍未過則停止並回報卡點 |
| 2.4 | **≥ 80 分**：把 high／low 發現逐項處理。**每項修正前必須實際開檔驗證審查者的說法為真**（審查者也會錯），驗證後才改 |
| 2.5 | 標示為「需 BA/TBS/後端確認」的項目**不得自行編造答案**，改為寫入開放問題表 + 施工閘門，並向使用者回報 |

### 第 3 段：詢問是否轉 PDF

| 步驟 | 動作 |
|------|------|
| 3.1 | 規格定版後，**主動詢問**使用者：「規格已定版（{分數}/100），要轉成 PDF 交付嗎？」 |
| 3.2 | 使用者同意 → 呼叫 `md-to-pdf` skill 轉檔（該 skill 支援中文、Mermaid 圖表、內嵌圖片） |
| 3.3 | 使用者不需要 → 直接結束，回報 spec.md 路徑與最終分數 |

---

## 輸出格式

流程結束時輸出：

```
## SD 規格產生完成

| 項目 | 內容 |
|------|------|
| 規格路徑 | {spec.md 路徑} |
| 版本 | {版本號} |
| 評分 | {分數}/100（{通過/有條件通過}） |
| 評分輪次 | {N} 輪 |
| 待外部確認項 | {數量} 項（見開放問題表） |
| PDF | {已產生路徑 / 使用者未需要} |
```

## 注意事項

| # | 事項 |
|---|------|
| 1 | **spec-writer 的預設規範與本 command 衝突**（它預設要 AC-XX、E2E 決策、獨立的手動測試步驟章節）。派工時不明講覆寫，產出就會走回預設格式 |
| 2 | **審查者的發現不可照單全收**。修正前一律開檔驗證，確認屬實再改；不實的發現要在回報中說明為何不採納 |
| 3 | **規格缺漏 ≠ 撰寫缺陷**。前者需外部確認，硬要 spec-writer 補只會生出更多推測內容，反而降低文件可信度 |
| 4 | 若需求文件本身有多個章節（如 BSD 的不同小節），**務必先確認哪一節才是目標系統的範圍**。把別的系統的規格誤當成本次範圍，是最容易發生且代價最高的錯誤 |
| 5 | 所有輸出使用正體中文 |
